Overview

Hi.
Generations is a personal project I began after taking an OOP course. The course gave me a taste of the power of good design. The illusionary dream of this good design inspired me to practice building a proper OOP program. Generations was also inspired by a previous personal project I did, called shoot-4-teams. You can see the repository here.

Anyways, what is Generations?

Generations is an application in which you can watch organisms of various species interact within themselves to simulate generations. Scalable in what usable "species" and "ecosystems" are.

Some vocabulary to know (coming up with these took a while. Read more about it in the development sections!).

  • Organism - A mutable object that can be observed and interacted with
  • Species - A description of a certain group of organisms
  • Generations - A group of organisms of the same species
  • Ecosystem - A description of how generations are advanced. Ecosystems define the type of species which are allowed in that ecosystem.

Cool, so how do I use Generations?

  1. Select a species.
  2. Select an ecosystem that the species can live inside.
  3. Generate an initial generation of organisms of the selected species.
  4. Advance generations depending on how the ecosystem defines advances.

Development

As expected, I spent a lot of time on paper trying to figure out the architecture. The first step was to figure out what to even call my objects. The term organism was fully defined only in the middle of development. The terms species and ecosystem were conceived in the middle of development. In shoot-4-teams, I defined a matchup as an organism, and a group of matchups as a generation. In Generations, I wanted to extend to more types of organisms. After that idea, I realized I could separate ecosystems into their own entities, and make them extendable. Back to the important semantics questions: What was an organism? What was a species? What was an ecosystem? Eventually, I settled on the definitions listed above in the overview. Although they may seem trivially understood, at the time it was a big semantics obstacle.

Notable Failures:

  • I wanted to allow for species behaviors to be easily compositely defined. For example, I wanted to take a description of a type of organisms which could fight other organisms called IFightingOrganism, and another description of asexual reproducers called IAsexualOrganism, then make a concrete species called AsexualFightingAlice which implemented both IFightingOrganism and IAsexualOrganism. I unfortunately found that Java did not allow multiple inheritance for concrete classes.
  • I wanted to be able to use class-level generic typing for ecosystems. This way, I would be able to ensure that, say Ninjas only fought other Ninjas, and this would allow for Ninjas to have a method called fight() which knew that it was fighting other Ninjas, instead of having to use typecasting. I ran into some terrible issues regarding Java wildcards, and had to scrap this feature.
  • I had terrible, terrible commit comments. I learned about git commit practices during the project.

Notable Successes:

  • I had clearly defined semantics, and stuck to them.
  • I had a proper seperation of concerns (Species did not need to know about generations, generations did not need to know about ecosystems) and encapsulation(ecosystems did not need to know how organisms worked, only what kind of behaviors they had).
  • I properly decoupled the model from the view using the MVC design pattern.
  • I documented regularly through the process both in writing and in code.

Reflection

Design is hard. There was a lot of trial and error although this was a relatively small system, and honestly it was difficult to gauge if I was coding too much or not coding enough. I would often find issues only after I had started coding, but I wondered if that was a sign that I had not thought through the design well enough.

Last semester, while I was taking my first OOP course, I talked about shoot-4-teams to a lot of recruiters. I discussed the ways that I had inadvertently applied OOP concepts before learning about them, such as abstraction and encapsulation (Disclaimer: with many, many, many mistakes and bad choices). I dove into Generations hoping to employ the same concepts in a more correct, powerful, and elegant way. I consider this project as a failure in terms of meeting those criteria. I have a long, long way to go. Nevertheless, Generations was a lot of fun and makes me proud to know that I learned a lot about design!
